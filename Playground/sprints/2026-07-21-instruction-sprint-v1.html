Goal

Primary goal: prove one large task works as one long conversation. A user starts a real multi-step task, the context window holds, agents hand off cleanly, and the voice never breaks — validated end to end, run multiple times, graded pass or fail.

To get there we work only on instructions and primers: the horizontal layers every agent shares (tone of voice, format rules, knowledge depth, the master instruction doc) and each agent's own instructions. Every agent flow gets validated and receives a golden standard: 20 to 50 binary pass/fail cases mined from real transcripts, not invented ones.

Hard rule: this is a quality sprint, not a feature sprint. No new agents, no model swaps, no architecture rebuilds. We read data before we write prompts, we change one thing at a time, and nothing merges without the suite passing.

Where we stand

What exists today

  • 14 product agents in ottavia-backend, each a static string in *.instructions.ts, composed with primers at runtime
  • Primer system in shared/primers.ts: 4 voice tones, 3 knowledge depths, methods, follow-up, data-pull contract
  • Golden samples + rubrics in eval/: voice.md, 14-dimension scoring.md, gold-standard replies per response type
  • Real eval harness in ottavia-engineering: Opus-as-judge grader, dashboard, trends with confidence intervals
  • Context strategy: prefetch-first briefs, 500-item history cap, cross-thread summary handoff

The gaps this sprint closes

  • Voice lives in 3+ places that drift: runtime VOICE_PRIMERS, eval/standards/voice.md, ops tone guide — with stale cross-references between them
  • No eval gates anything. The harness runs by hand against Mongo dumps; a prompt change can ship with zero automated checks
  • Patch-on-patch bloat: Nutrition at 230 lines of MANDATORY/CRITICAL rules; the known debt in instruction-reduction-plan.md is diagnosed but unpaid
  • Routing rules duplicated in prose between Gateway (136 lines) and the newer Classifier — two sources that can diverge
  • No versioning discipline: INSTRUCTIONS_VERSION frozen at "v1"; layer order copy-pasted per agent, so a layer can silently go missing
Method, borrowed from the best teams
Error analysis before edits
Read ~100 real transcripts, code failures into categories, count them, fix the biggest first. The best teams spend 60 to 80% of prompt time reading data, not writing prompts. No prompt edits days 1 to 3.
Hamel Husain · Lenny's interview
Golden = binary, from real failures
20 to 50 cases per agent drawn from actual bad transcripts, never hypothetical. One-sentence pass criterion two people would verdict identically. No Likert scores.
Anthropic · Demystifying evals
Single-source layered primers
Tone, format, and safety live in one composable file each, assembled into every agent at build time. One place to edit, zero drift. Prompts written at the right altitude: heuristics and canonical examples, not edge-case laws.
Sierra Agent OS · Anthropic context engineering
Prompts as code, suite as gate
Every instruction change is a PR with an eval run attached. Regression suites held at ~100% pass; capability evals drive improvement and graduate into regression. "Eval-driven development is the new TDD."
Notion × Braintrust · industry standard
Long runs graded pass^k
Scripted long-session scenarios recorded in full, replayed from clean state 3 to 5 times; all runs must pass. Failures bisected to the first upstream error, replayed standalone before touching a prompt.
Anthropic · Hamel Husain
Deterministic first, judge second
Cheap programmatic checks (format present, length, banned phrases) run first; the LLM judge handles tone and empathy only after being calibrated against 30 to 50 human-labeled transcripts.

Fact stated early, needed late. Behavior across the 500-item history cap and the cross-thread summary handoff. Does prefetch-first actually hold the thread?

20
Seamless handovers
Agent switches must be invisible: one voice throughout, no tone or format reset when routing changes, never re-asking something already said, no "let me connect you" seams. Rubric: reading the transcript, you cannot tell where one agent ended and the next began.
Voice primer across all agents · handoff golden cases
21
Multi-intent messages
Two tasks in one message: "log my lunch and update my goal", "how did I sleep, and remind me to stretch". Both intents detected, both executed, answered as one coherent reply, nothing dropped. Golden set of 2-task combos across agent pairs.
classifier.ts routing · pairwise intent combos
22
Fix loop
Every failure bisected to the first upstream error, replayed standalone as a single turn, fixed with one change, re-run. New failures become permanent golden cases.
First-failure annotation · one change per PR
Why this is the primary task. Everything else in the sprint exists to make this pass: primers keep the voice stable across agents, per-agent golden standards keep each handoff correct, and the eval stack is how we know instead of hope. Gate 2 is a live run of this, end to end.

Workflow, order of edits is non-negotiable
Transcripts → Golden standard → Instruction edit → Suite run → PR + deploy
Transcripts first
No instruction is edited before its agent's real failures are read, coded, and counted. The biggest category gets fixed first. Days 1 to 3 are reading only.
Golden standard before rewrite
The target is written down before the prompt changes: binary cases with one-sentence pass criteria. If Vika and Elad verdict a case differently, the criterion is rewritten, not argued.
One change at a time
Each PR changes one thing: one section, one rule, one primer. Deletion counts as a change and is a first-class deliverable, every prompt gets a token budget.
Suite gates the merge
Regression cases stay at 100% pass. A red suite blocks the PR, no exceptions, including Vika's own edits.
Out of scope, scope guard
  • New agents (psychology, concierge, correlation, weather stay in planning)
  • Model or provider swaps, tier changes stay as they are
  • Runtime prompt store or DB-backed prompt editing
  • Gateway architecture removal, we dedupe rules, not rebuild orchestration
  • Mid-thread summarization or memory system rebuild, we test the boundary, we don't redesign it
  • UI, DESIGN.md, and the future-design branch, separate workstream
  • Marketing tone guide rewrite, it only gets aligned to the single source
  • Expert knowledge / methods MVP content, instructions only touch how experts render
